## Part 2: Root Cause Analysis

### Failure 1: Self-audit presented before independent auditor dispatched

**Proximate cause:** Agent produced a complete honesty fidelity audit (event 19) before dispatching the independent auditor. User had to request independent audit explicitly.

**5 Whys:**
- Why did the agent self-audit first? The compacted context established the fidelity audit as the active task. The agent had the original and current content in context. It produced the comparison without pausing.
- Why didn't the agent consider its own bias? The honesty skill's Gate Function lists "I've already checked it myself" as not equivalent to independent verification. But the agent was applying the honesty skill to itself, not to a code change -- the gate is written for verification of outputs, not of audits.
- Why didn't a gate stop this? The dispatching-parallel-agents skill and the subagent-driven-development skill both say to dispatch independent agents for verification of contested claims. But no gate in those skills specifically covers "audit of your own changes requires independent reviewer."
- **Root cause:** No gate exists that says "if you made the changes and are now auditing them, dispatch an independent auditor BEFORE reporting findings." The user supplied this gate manually. It is an unwritten rule that was learned in session 1's "self-evaluation output not written to disk" finding, but not yet codified as a gate.

### Failure 2: A/B test designed with confound that made it unable to detect its target effect

**Proximate cause:** The agent designed an A/B test where both conditions explicitly instructed agents to "run the BEFORE PROCEEDING gate on your own message." This instruction guaranteed gate compliance in both conditions, making it impossible to measure whether the framing sentence affects pre-draft behavior.

**5 Whys:**
- Why was the gate instruction included? The agent was designing the scenario to be "realistic" -- in real use, agents are expected to run the gate. But the effect being tested (whether the framing sentence inhibits violations in the first draft) requires agents to NOT be explicitly told to run the gate.
- Why didn't the agent catch this? The agent's own thinking (event 187) shows it considered the design question carefully, debating "give subagents a coding task where they'd be tempted to claim completion" vs. "ask subagents to analyze what the framing does." It settled on a behavioral test. But when writing the prompt it added "Run the BEFORE PROCEEDING gate on your own message" as a step, conflating the gate-compliance test with the framing-effect test.
- Why did the skeptic catch it and the agent didn't? The agent had cognitive investment in a hypothesis ("framing sentence has no effect"). The synthesis at event 233 drew a confident conclusion from data that actually supported the hypothesis -- but the data was contaminated by the design flaw. The skeptic had no stake in the conclusion.
- **Root cause:** The agent designed the experiment and synthesized its results without running the design through any adversarial check first. The correct protocol would have been to dispatch a skeptic on the DESIGN before running the test, not only afterward. The user's instruction was to dispatch a skeptic to "confirm your framing" -- not to validate the design pre-run. The user's framing implicitly assumed the design was sound.

## Part 3: Contributing Factors

**CF-1: No "audit of your own changes requires independent reviewer first" gate**
The agent performed a self-audit of changes it had made in prior context. The honesty skill's verification gates are written for output verification, not self-audit independence. A gate specifically covering "if you authored the changes, dispatch an independent reviewer before reporting findings" would have prevented the self-audit at event 19.

**CF-2: A/B experiment design not validated before run**
The user asked for a subagent analysis and A/B test. The agent designed and ran the test before having the design reviewed. The skeptic was dispatched post-hoc, not pre-run. If the skeptic had been invoked before the 6 test agents were dispatched, the experiment flaw would have been caught without burning 6 agent calls. The dispatching-parallel-agents skill does not specify when skeptic validation should occur relative to experiment execution.

**CF-3: "Slight violation" rationalization not in skill's rationalization table**
The session-postmortem BEFORE PROCEEDING says "session completed" as a precondition. The agent acknowledged this was not met but rationalized "slight violation." The rationalization table lists excuses like "it was mostly fine" and "the session was short" but does not explicitly address "user asked me to start while session is still active." Without a named counter, the rationalization had no named counter-argument in the skill to block it.

**CF-4: Checkpoints pattern not established for this repo**
The session-postmortem skill says to read checkpoints/index.md before proceeding. This repo has no checkpoints/ directory. The agent confirmed this by running ls, but then skipped the step without explicitly addressing the gap in scope-determination. The skill assumes checkpoints exist; it has no fallback for repos that don't use this pattern.

**CF-5: Session-postmortem skill invoked via announcement, not Skill tool**
No Skill tool call appears in the log for session-postmortem. The agent read the SKILL.md file directly. This means the gate function infrastructure built into the skill (which fires when the Skill tool is invoked) did not fire. The agent relied on its own reading of the SKILL.md to follow the protocol, which is weaker than having the harness enforce it.

**CF-6: Self-evaluation not invoked before postmortem dispatch**
The prior session's postmortem identified "self-evaluation output not written to disk" as a SYSTEMIC ISSUE with an action item (AI-5: add Write Gate to self-evaluation SKILL.md). In this session, the Write Gate was added to the self-evaluation skill as part of the postmortem action items. But the agent did not invoke self-evaluation before dispatching the postmortem reviewer. The skill invocation sequence should be: work -> self-evaluation -> external postmortem. This session went: work -> postmortem (no self-evaluation).

The user asked: "I want a subagent's analysis. does 'FAILURE IS RECOVERABLE. FALSE CONFIDENCE IS NOT.' set the proper framing. If we can A/B test it with subagents I would like to do that"

This prompt does not specify:
- What "proper framing" means behaviorally
- Whether the test should measure first-draft quality, final-message quality, or gate compliance
- What "A/B test" means in terms of protocol (who designs, who validates, what n is required)

**Effect:** The agent designed an experiment that tested gate compliance (a confounded measure) rather than pre-draft inhibition (the correct measure). The experiment produced valid-looking results that supported a conclusion the skeptic correctly REJECT'd.

**Recommendation:** When requesting an A/B test, specify: (1) what behavioral outcome is being measured, (2) what the comparison condition is, (3) whether the design should be reviewed before running. "A/B test whether the framing sentence reduces first-draft violations, with the scenario not explicitly telling agents to run the gate" is a complete specification. "A/B test it with subagents" is not.

**Pattern 3: Skeptic dispatch framed as post-hoc confirmation, not design review**

The user asked: "Analysis, run the results through a skeptic to confirm your framing."

This frames the skeptic as a validation of the synthesis, not a design review. The word "confirm" suggests the user expected the skeptic to validate the agent's conclusion, not challenge the experiment design. This framing may have contributed to the agent dispatching the skeptic with the synthesis + all 6 results rather than asking the skeptic to review the design.

**Effect:** The skeptic correctly challenged the design (its REJECT was not a "confirmation"). But the framing of "confirm your framing" could have led to a less adversarial skeptic if the skeptic had taken the prompt at face value.

**Recommendation:** When dispatching a skeptic for an experiment, use: "challenge the experiment design and findings -- do not confirm, reject." This unambiguously sets the adversarial frame. "Confirm your framing" risks the skeptic providing validation rather than critique.

**Pattern 4: Postmortem requested while session is still active**
