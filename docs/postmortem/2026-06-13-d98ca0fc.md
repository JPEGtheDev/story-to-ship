### Root Cause Analysis

#### Failure 1: Implementation committed directly to main

Why was code committed to main?
  -> No feature branch existed when implementation started.
Why was no feature branch created before implementation?
  -> No gate in session-bootstrap, execution, or writing-plans requires "create and checkout a feature branch" as a precondition before the first file change.
Why did the session-bootstrap BEFORE PROCEEDING gate not catch this?
  -> The gate says "check git status" but does not require a branch creation step as output of that check.
Root cause: No structural gate enforces branch-before-code. The rule exists conceptually but is not encoded as a STOP condition anywhere in the skill chain.

#### Failure 2: Feature branch created but never checked out

Why was the branch not checked out after recovery?
  -> The recovery sequence used `git branch <name> <hash>` (creates branch at commit) then `git reset --hard HEAD~1` (moves main back). This sequence leaves the working tree on `main`.
Why was no `git checkout` added to the recovery sequence?
  -> No explicit todo step existed for "checkout the new branch." The recovery protocol's goal was "preserve the work + restore main" -- not "land on the feature branch."
Root cause: The recovery protocol is incomplete. It restores the commits to a branch but does not move the working tree to that branch.

### Where We Got Lucky

1. **`git reset --hard HEAD~1` succeeded cleanly.** The main branch had exactly one commit ahead of origin. If there had been multiple commits, a diverged remote, or staged changes, the recovery would have required cherry-picking or a rebase -- much higher risk of data loss. The correct outcome was reached, but through a risky path that depended on the branch being exactly one commit ahead.

2. **No one else was on the repo during the broken state.** The window where main had the undisclosed implementation commit could have been observed or pulled by a collaborator. The user caught it before any downstream use.

3. **The fix agent's grep pattern over-corrected** (changed "workflow and rules" -> "process steps and rules" because `workflow` matched the exclusion grep), and Stage 2 called it benign. If the over-correction had introduced a meaningful semantic change, it would have shipped undetected.

### Prompt Feedback

**Pattern found:** Scope addition after implementation starts
**Example:** "we can keep /docs for this update" -- added mid-brainstorming after the scope conversation had progressed to DDD taxonomy design.
**Effect:** Required a mid-brainstorming adjustment. Low cost in this case, but established a pattern of scope clarification arriving after design work had started.
**Recommendation:** Before brainstorming begins, confirm the full scope (including what stays unchanged like `/docs`) in a single pre-brainstorming checklist. This surfaces "keep X" decisions before design rather than during it.

**Pattern found:** Approval mid-execution
**Example:** "any remaining items or are we good to execute?" -- phrased as a binary gate but timing was ambiguous (Skeptic review was not yet complete when this was asked).
**Effect:** Created ambiguity about whether approval covered the pre-Skeptic plan or the post-Skeptic revised plan. The correct resolution (wait for Skeptic, re-present revised plan) was followed, but the prompt pattern creates a risk of bypassing the Skeptic gate.
**Recommendation:** Approval prompts should specify what state is being approved: "approve the post-Skeptic revised plan?" rather than an open-ended readiness check.

## Addendum: Skeptic Agent Worktree Self-Check Failure

**Session:** d98ca0fc (continuation, docs/documentation-skill-c2-grounding PR review phase)
**Date:** 2026-06-10

### What Happened

A Skeptic Agent was dispatched to review the SKILL_DISPATCH_TABLE.md deletion decision. The agent template contains this worktree self-check:

```bash
git rev-parse --show-toplevel
```

The agent returned BLOCKED:
```
STATUS: BLOCKED
Not running in the expected worktree. `git rev-parse --show-toplevel` returned /home/JPEG/Projects/story-to-ship,
expected /home/JPEG/Projects/story-to-ship/.worktrees/agent-skeptic-dispatch-table.
```

The agent was then continued via SendMessage with instructions to use absolute worktree paths for all file reads. It then completed successfully.

### Root Cause

The Agent tool always dispatches subagents starting in the main repository working directory (`git rev-parse --show-toplevel` == repo root). `git rev-parse --show-toplevel` without `-C` always returns the main repo path regardless of what WORKTREE_PATH is passed in the prompt.

The `skill-reviewer.md` template uses `git -C {{WORKTREE_PATH}} rev-parse --show-toplevel`, which correctly verifies the worktree path by specifying the target directory explicitly. The `skeptic.md` template lacked the `-C` flag, causing a guaranteed false BLOCKED on every dispatch.

### Impact

1. First Skeptic dispatch blocked. Required a SendMessage workaround.
2. The SendMessage workaround (telling the agent to use absolute paths) worked but is fragile -- it relies on the agent interpreting informal instructions rather than following a structured template.
3. Any Skeptic dispatch in any session before this fix would have failed the worktree check.

### Fix Applied

`agents/skeptic.md` worktree check updated from:
```bash
git rev-parse --show-toplevel
```
to:
```bash
git -C {{WORKTREE_PATH}} rev-parse --show-toplevel
```

### Pattern: Worktree Check Template Consistency

All agent templates that include a worktree self-check MUST use `git -C {{WORKTREE_PATH}} rev-parse --show-toplevel`. Templates using the bare `git rev-parse --show-toplevel` will always fail when dispatched via the Agent tool (which starts in the main repo context).

**Audit action:** Check all `.claude/agents/*.md` files for bare `git rev-parse --show-toplevel` and replace with `git -C {{WORKTREE_PATH}} rev-parse --show-toplevel`.
