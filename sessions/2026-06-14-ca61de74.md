### Root Cause Analysis

**Failure 1: Wrong branch selected after "check out a working branch"**

Why did the agent check out `claude/greenfield-app-skills-nx9Q4` instead of branching from `main`?
  Because the agent's THINKING (event 211) pattern-matched "existing greenfield feature branch" to "this is the right base for greenfield work."
  Why did the agent pattern-match without verification?
    Because neither `session-bootstrap`, `using-git-worktrees`, nor `three-amigos` contain a rule requiring `git log main..HEAD` or `git merge-base` verification when selecting a branch to work from.
  Why is that rule absent?
    Because no previous postmortem explicitly identified "existing named branch assumed current" as a rationalization pattern and added a Red Flag to the relevant skills.
  Why was the remote branch visible (event 41: `remotes/origin/claude/greenfield-app-skills-nx9Q4`) interpreted as a valid base rather than a historical artifact?
    Because the agent read the name as semantic signal ("greenfield app skills") without reading the commit graph to compare against `main`.

**Root cause:** No gate in any loaded skill required checking `git log main..HEAD` or `git merge-base main HEAD` before selecting an existing branch as a working base. The decision was made by name recognition alone.

**Failure 2: Agent presented skeptic verdict as structurally valid after running on stale base**

Why did the agent relay the skeptic's findings without disclosing that the worktree ran against a stale branch?
  Because after checking out the wrong branch, the agent stopped tracking "this branch is not main-current" as a live concern.
  Why did it stop tracking?
    Because there is no rule that requires base branch provenance to be included in subagent output synthesis.
  Why is that rule absent?
    Because the using-git-worktrees skill documents how to create and verify worktrees but does not require the orchestrator to verify that the current branch is at the correct base before dispatching.

**Root cause:** The skeptic finding about greenfield-discovery was structurally contaminated by the stale base but was relayed as APPROVE_WITH_CONDITIONS without disclosing the base branch state. False confidence claim: agent treated a stale-branch artifact as a valid finding (event 373, 04:01:59).

**Failure 3: Premature skill edit during self-evaluation**

Why did the agent apply an edit to `session-bootstrap/SKILL.md` at 04:15 before completing self-eval Steps 1-5?
  Because the self-evaluation skill's BEFORE PROCEEDING gate says "no further implementation work is planned for this session" and the agent had not yet verified that condition (branch switch was still pending, AC updates were still incomplete).
  Why did the BEFORE PROCEEDING check not block the edit?
    Because the agent did not run the BEFORE PROCEEDING checklist explicitly -- it jumped directly to Step 6 (Apply Updates) after identifying the fix target.
  Why did the skill dispatch not enforce this?
    Because self-evaluation BEFORE PROCEEDING is a checklist, not a structural gate. The agent can read the checklist and still act on Step 6 without having verified Steps 1-5 are complete.

**Root cause:** Self-evaluation BEFORE PROCEEDING is advisory text, not a structural gate. The agent's pattern of "I see the fix, apply immediately" bypassed it.

### Where We Got Lucky

**Lucky outcome 1: Amigo NEEDS_CONTEXT verdict masked the stale-base problem.**

All three amigos returned NEEDS_CONTEXT -- not a verdict that depended on repo file content. If any amigo had returned APPROVE or REJECT based on reading skill files, the stale-base contamination would have produced a structurally incorrect vote that could have been acted on. The fact that NEEDS_CONTEXT doesn't depend on repo file inspection means the stale state did not poison the amigo output. The Three Amigos ceremony produced a valid result by luck, not by correct process.

Gate that would have caught it: a BEFORE PROCEEDING check in `using-git-worktrees` requiring `git log main..HEAD` to return empty before any subagent dispatch with repo-read responsibilities.

**Lucky outcome 2: Issue updates were made to GitHub, not to repo files.**

Between the Three Amigos ceremony and the skeptic dispatch, the agent updated Issues #4 and #5 on GitHub. These are remote state changes (gh issue edit), not repo file changes. Because no repo files were modified while on the stale branch, there is nothing to revert. If the agent had written to any skill files during this period, those changes would have been on the wrong branch and required cherry-pick or manual correction.

Gate that would have caught it: same as above -- base branch verification before any work begins that would produce artifacts.

**Lucky outcome 3: The user interrupted the gh issue edit at the right moment.**

At 04:02:34, the user rejected the `gh issue edit 5` call at the exact moment the agent was about to modify Issue #5 based on the skeptic's APPROVE_WITH_CONDITIONS result (which included a finding from the stale branch). The finding about greenfield-discovery not existing would have produced an incorrect issue update. The interrupt prevented a misleading change. If the user had not interrupted, the issue would have been updated with incorrect cross-reference information.

No gate exists for this. The only mitigation would be explicit base branch verification before relaying skeptic verdicts.

### Prompt Feedback

#### Pattern found: Implicit requirement treated as obvious

**Example (quoted):** "first you need to check out a working branch" (event 209, queued_command)

**Effect:** The instruction names the action category ("check out a working branch") but not the specific branch, not whether to create a new one, and not from which base. The agent had exactly one branch visible in context that matched "working branch for greenfield work" -- `claude/greenfield-app-skills-nx9Q4` -- and used it. The implicit requirement ("from main, fresh") was unstated.

**Recommendation:** When directing branch operations, state the full intent: "create a new branch from main for this work" rather than "check out a working branch." The distinction between "use existing" and "create new from main" is the exact decision the agent failed to make correctly. Making it explicit eliminates the decision.

#### Pattern found: Approval mid-execution creating ambiguous scope

**Example (quoted):** At event 209, user sends "first you need to check out a working branch" as a queued_command while the agent had just completed `git check-ignore` and was about to create worktrees. The instruction arrived mid-worktree-creation-sequence and was processed as an additional step before, not as a redirect to a different branch strategy.

**Effect:** The agent interpreted the instruction as "you forgot a step" (adding a checkout) rather than "stop and rethink the base branch." The mid-sequence injection was processed additively, not as a constraint on the entire upcoming sequence.

**Recommendation:** When redirecting mid-execution, explicitly signal the scope of the redirect: "before creating the worktrees, make sure you're on a new branch from main." This makes the intent a constraint on the sequence, not just an extra step.
